This page gives you a short overview of the server's source code.

HellgroundCore

Directory src/hellgroundcore contains the main program of project, the place where real server starts. This part of core consist of:

Main
Entry point of the program, the int main(...) function is here. This is where the core processes input arguments, decides in what mode should it run (service for Windows, daemon for *nix or just simple casual program) and then wakes up the Master.
Master
A singleton class which prepares whole server to run. It:
  • connects us to database;
  • initializes world settings (reads config file);
  • catches interrupt signals for program;
  • starts command line thread;
  • sets process priority and processor affinity for world thread;
  • launches freeze detector;
  • starts network socket listener.
After core finishes its work, it also closes database connections and do some clean-up before the program exits.
CLI Runnable
The mighty console, this is the user interface for server administrator. Also here we have implementations for console-only commands like account creation/deletion.
World Runnable
World thread, the void WorldRunnable::run() is the every heartbeat of game world. From here the world update function is called every 50 miliseconds.

HellgroundRealm

Directory src/hellgroundrealm contains code for the realm list server. This is the program to which players connect first, it authorizes users and then gives them all necessary info about realms. Then the player chooses one of realms and connects to its HellgroundCore program. The structure of realm server is as follows:

Main
The program entry point, the int main(...) function. Unlike HellgroundCore, there is no Master class, so all initialization stuff is done here. This is because HellgroundRealm is much simpler, it does not need any thread chewing data all the time (like World updates in the Core). It just needs to respond to incoming connections.
  • processes input arguments, chooses mode (just like the Core main function);
  • opens config;
  • starts database connection (only accounts database is needed here);
  • cleans up expired bans;
  • starts network socket listener.
AuthSocket
Class which handles a client connection. Every time a player connects to HellgroundRealm, an object of this class is created for him to process the connection. Here we have functions for each of the possible client commands that can be sent to realm server. Those functions handle all authorization of user using the SRP6 protocol. They also check if the user is banned, what version of client he has, and what operating system he is using. After authorization process is completed, HellgroundRealm sends realm list to the client and saves all necessary data to characters database, so that it can be used by HellgroundCore.
RealmList
Class which stores and updates realm server's information about running realms. It periodically queries database for information about the running servers.
TOTP
Time-based One-time Password Algorithm implementation. It gives us the possibility to increase account security, as the connection requires not only a normal password but also second one, which is generated using secret key and current time.

HellgroundScripts

Scripts is the directory storing various in-game scripts. They are mostly used to determine the behaviour of NPCs and objects. HellgroundScripts are built as external library and the core calls every script only via few functions. This mechanics is very rarely modified, because it works. So for each developer most important part are the scripts themselves.

Scripts sorting

Basically all scripts are sorted by zone they are connected to. There are few exceptions:

Spell scripts
Some spells are scripted, that is because making exceptions of exceptions of exceptions in core would be hard to implement and problematic in debugging.
Area trigger scripts
Scripts triggered when a player enters specified area, for example teleports other than simple instance entrances with more complicated triggering.
Item scripts
Generally there are quest items with some complicated requirements to activate. Some of scripts are used to prevent abusing of such items.
Commonly used scripts
Guard script, innkeeper script, profession master script etc.
Trans-zone scripts
Scripts for NPCs that can appear in many places, like non-combat pets.
Script structures
Those are not scripts in fact, just abstract structures to be filled with AI, eg. ScriptedAI or BossAI. All AI scripts are derived from them.

Script files

Each script file consists of:

  • header with licence info;
  • includes, we always #include "precompiled.h" which gives us nearly everything we need to script. Sometimes we additionally include a header with some definitions (eg. when scripting boss we need some definitions of instance script);
  • script structures and functions, more about them later;
  • AddSC_...() (add scripted creature) function, this is an important one. This function registers scripts in script manager, here we connect script functions with script names, every such function is called when loading scripts in ScriptLoader.cpp.

Script functions

Script manager mechanics basically connects a script name (like 'npc_some_dude') to some functions (like function defining what our dude will do if we talk to him). Most common functions are:

Gossip hello
NPCs reaction to talking, here we usually add some options to the talk menu.
Gossip select
NPCs reaction to selecting an option from the talk menu (eg. NPC starts walking).
Quest accept
What happens after we accept quest from this NPC/object (eg. start escort).
Use gameobject
What happens after activating the object.
Get instance data
For use with instance scripts. An instance script is based on the ScriptedInstance structure. We first define a child structure like struct instance_instancename : public ScriptedInstance and Get instance data function is used to create new instances of this structure.
Get AI
For NPC scripts, if the NPC is supposed to have sophisticated AI (eg. most bosses), we define an AI structure struct boss_some_one_AI : public ScriptedAI, then create GetAI functions which returns us new boss_some_one_AI(...).

(Refer to the definition of Script in ScriptMgr.h for full list)

game

This is the heart of the project. Game directory contains all rules of our game world. There is all kind of stuff here, so you'd better not try to understand everything at the beginning. Let's run quickly over major parts:

Anticheat
Speed hack, fly hack, water walk hack and others can be detected and this is the place where it is done.
Battlegrounds
Battleground definitions along with the battleground manager. This is where battlegrounds queues are handled and where we define rules for a battleground.
Chat commands
Every in-game command, from ".account announce" to ".wp unload" has its function, here they are stored.
Handlers
Server reaction to data from client. Every packet sent by (and to) the client starts with its opcode, which tells us what this packet is about. Is it a chat message or logging out query, after reading an opcode, the server redirects data to appropriate methods called handlers.
Lua Engine
Our server has implemented Lua scripting language interpreter (based on Eluna project).
Maps
A map is a 'geographic' class. Kalimdor is a map, Eastern Kingdoms is a map, Hellfire Ramparts are maps, many maps. If two players are in instance on two different IDs, they are on two different maps — with the same map ID (eg. 543 for Ramparts) but with different instance ID (this is the ID you see in-game). It may seem complicated but in fact it is not, really.
Movemaps/vmaps
Hard stuff here, line of sight and movement calculations.
OutdoorPvP
Towers in HP, Haala in Nagrand etc. Zone-wide PvP objectives are here.
Server
Client-server communication.
Spells/skills
Each player's or npc's ability except meele attack is considered a spell. Important place is SpellMgr::LoadSpellCustomAttr() (SpellMgr.cpp) where we override spells' data stored in dbc.
Warden
Our good friend, detects all kinds of cheats.

Artificial Intelligence

Every unit in game has its AI which determines behaviour (except when controlled by player). Here we have definitions of base classes and most common AIs. Process of selecting AI is defined in CreatureAISelector.cpp, it can be determined based on various conditions. Some AI classes we have in the core:

UnitAI & CreatureAI
Those are the basic AI classes, they are not used in game, but are parent for all other AIs. UnitAI contains declarations of virtual functions (eg. virtual void UpdateAI(...)) and definitions of helper methods used for interaction with other units (eg. std::list<Player*> FindAllPlayersInRange(...)). CreatureAI is for units that fight, it contains UpdateVictim(...) function which handles choosing target to attack. It also has declarations of many virtual functions which are called as a result of combat event (like virtual void MoveInLineOfSight(...) or virtual void OnSpellHit(...)).
ScriptedAI
This is a class for special AIs, defined in sc_creature.h. More about scripted AI in the scripts section.
AggresorAI & CasterAI
Two simple AIs for common NPCs. Aggresor simply fights with melee weapon, Caster additionally casts some spells assigned in the creature template.
PetAI
Class for players' pets, it has additional functions helping with autocast settings.
PlayerAI
Those are AIs used when the player is mind-controlled by an NPC, each player class has its own AI class (WarriorAI, DruidAI etc.).
EventAI
This is useful AI class, it creates the possibility to determine some NPC's behaviour based on database entries. Gives us more possibilities than standard AIs but not as many as ScriptedAI. Its large advantage is being based on database, not on code, so it can be modified and reloaded without restarting of the server.

Objects

An object is any part of game world which changes in time. Player, item or NPC is an object. Main classes starting from the bottom:

Object
Most basic class, all other objects are derived from it, but an Object itself does not appear in-game. Basicaly an Object is a set of 32-bit values usually called fields(which can be interpreted as uint, int or float) and some methods used for manipulating those values as well as sending them to client. In UpdateFields.h you can find aliases of fields, each kind of objects has its own set of fields. Only fields that are common to all Objects are its GUID(Global Unique Identifier), Type and Entry (number of object template).
WorldObject
Also only a parent class. This one is for objects that appear in game world, simply objects that have their position (all objects except items). It stores a little more information - x,y,z position, orientation (angle) and map on which object exists. It also has some functions which handle this additional data - relocating, calculating distance and line of sight.
Unit
Another only parent class. Here we have WorldObjects that can be selected, this means players, NPCs, pets etc. This class has much much more to handle than WorldObject. An Unit has HP,buffs, debufs, enemies, can cast, move, attack, be attacked or interact with player in other way (can be a player too). This gives us more data to store (check enum EUnitFields in UpdateFields.h) and more functions to handle them. Unit class is where we calculate damage done by attack or spell, handle being friendly or aggresive, cast spells or modify stats.
Creature
First of classes that do appear in game. Creature is for all units except players, this means NPCs, pets and totems. It has its loot and threat, can be a vendor and respawn. It has functions which handle interaction with players, respawning and looting. Each Creature has its AI.
Player
This is class which represents player's avatar in game. Contrary to other units it is controlled by external commands not AI, it has equipment, skills. This class also handles mailing, quests, PvP, RaF, battlegrounds, instancing, groups... and lots other things.
GameObject
Class for 'clickable' WorldObjects. Doors, chests, buttons... (there are 34 types of them in total). Some of them can be looted, respawn or trigger spells. There is a wide variety of them and basicaly what a GameObject does depends on what type it is.
Item
What an item is every player knows. This class handles being soulbound, conjurable or a container. Some items have uses or can start quest. You should remember that every instance of this class is represented in game by one icon (if you have stack of arrows it is one item). All items that are 'of the same kind' are multiple Items, but have the same ItemPrototype.

shared

The shared directory is the place where we hold parts common both to HellgroundCore and HellgroundRealm.

Auth
Cryptographic classes, used in data encryption and user authorization. The interface between core and OpenSSL.
Config
Configuration file opener and interpreter.
Common.h
A header with definitions common both to core and realm.
Database
Classes which handle connection to the database, interface between core and MySQL. Here we also hold DBC files loader.
Log
Logging tool.
Util
String format converters, some random number methods.
Wheaty Exception Report
Crash log generator for Windows.

others

There are many other elements that don't have a place in any of projects' 'main branches' but without them, the server won't work at all.

genrevision

Genrevision is a simple tool used to determine actual revision of server's repository. It is capable of extracting data from Mercurial (default), Git or SVN files. When genrevision is executed, it creates revision.h file which stores info about the current code version. It should be always built and run before compiling any other part.

framework

The framework directory contains a few useful parts for the HellGround machine.

Platform
Files where platform- and compiler-dependent defines are stored.
Policies
This is the place where our memory allocation functions are defined.
Dynamic
Headers for the factory pattern, we use factory for creating AI for NPCs.
GameSystem
This is home of grids. A grid is a basic segment of the game world, a simple sector of Warcraft reality.
Utilities
Here we have utilities, few classes and methods which simplify parts of the code, like endianness converter or unordered map class.

dependencies

ACE (Adaptive Communication Enviroment)
A toolkit we use to simplify networking and threading. This is the part which handles connections from the clients and helps to prevent data races between multiple threads of the core.
TBB (Threading Building Blocks)
Another toolkit used in the core. This one gives us memory allocation and cooperates with ACE in handling threads.
Recast, Detour, G3D
Those are libraries used for handling three-dimensional stuff, VMaps and MMaps are based on them.
zlib, bzip2
Basically data compression stuff.
OpenSSL
Cryptography toolkit, used when authorizing users, also connections with clients are partially encrypted.
MySQL
Connects our core to the SQL database.
Mersenne Twister
The 623-Dimensionally Equidistributed Uniform Pseudo-Random Number Generator. We need to be a little randomness sometimes.
LuaLib
Lua language engine based on the Eluna project. Thanks to it, the core can process Lua scripts.

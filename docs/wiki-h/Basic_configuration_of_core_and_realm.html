In the introduction, we explained that there are two major applications that make up your server software. Each of these has a configuration file that allow you to tailor the way your server works. Although the system allows a considerable amount of customization, this article will explain only the settings that are required to start the server for the first time.

After running the server, you may want to change various other aspects of the configuration. Each section has an extensive list of the options, and possible settings in most cases, that will be available.

Let's begin with the authentication server, hellground-realm.

hellgroundrealm.conf

Before we can begin, we have to make a working copy of the configuration file. The installation process has provided an example, hellgroundrealm.conf.dist. You can find this in your server directory on Windows, or in your $HOME/etc directory on Linux.

Make a copy of the file, and call it hellgroundrealm.conf. This will be our actual working configuration.

Initially, your configuration options will look like this: 

LoginDatabaseInfo = "127.0.0.1;3306;hellground;hellground;realmd"
LogsDir = ""
MaxPingTime = 30
RealmServerPort = 3724
BindIP = "0.0.0.0"
PidFile = ""
LogLevel = 0
LogFile = "realmd.log"
LogTimestamp = 0
LogFileLevel = 0
LogColors = ""
EnableLogDB = 0
DBLogLevel = 1
UseProcessors = 0
ProcessPriority = 1
RealmsStateUpdateDelay = 20
WrongPass.MaxCount = 0
WrongPass.BanTime = 600
WrongPass.BanType = 0

While there are a lot of options here, the only required change is to set the LoginDatabaseInfo line. The format of this line is: 

    DATABASEHOST;DATABSEPORT;DATABASEUSERNAME;DATABASEPASSWORD;DATABASENAME
  • If your database is running on the same server as your core, the DATABASEHOST can be left as 127.0.0.1. If your database is on a different machine, change this to the hostname of the database server.
  • The DATABASEPORT is the port that your DB server is set to answer on. For MySQL, the default is 3306. Unless you have changed your MySQL server configuration, the default can be used here (3306).
  • It is very good practice to create a new database user for each application that uses the database. By default, MySQL will create a user called root, and prompt you for a password when you install the database server. While you can use root as your DATABASEUSERNAME, you really should set up a special user for HellGroundCore. Regular database-type geeks will laugh at you and call you noob if you use root.
  • The DATABASEPASSWORD is, of course, the password for our DATABASEUSERNAME.
  • Last is the DATABASENAME. Unless you have decided to use a different name for some reason, the authentication database name should be realmd.

Linux users will probably want to set a few more options to keep their $HOME directory in good order. Suggestions are:

  • Set your LogsDir to $HOME/log, where $HOME is the full path (/home/hellgroundcore/log, for example).
  • Set your PidFile to realmd.pid

Two of these options will only apply on the Windows platform:

  • UseProcessors, which sets the processor affinity. Unless you have done some serious studying of the Windows infrastructure, this does not do what you think it does. If you don't know what processor affinity is, leave the default. If you want to learn about processor affinity, and change it later, an internet search will give you some of the information you will need. Explaining it here is beyond the scope of this document.
  • ProcessPriority which sets the priority level of the server process. We don't want any latency, if we can avoid it, so leave this at 1.


hellgroundcore.conf

Now let's take a look at the configuration for the world server, hellground-core.

Before we can begin, we have to make a working copy of the configuration file. The installation process has provided an example, hellgroundcore.conf.dist. You can find this in your server directory on Windows, or in your $HOME/etc directory on Linux.

Make a copy of the file, and call it hellgroundcore.conf. This will be our actual working configuration.

The number of options in the hellgroundcore.conf file make it impractical to show the whole file here. You will need to search for the various required settings in the file as we continue.


The most important changes to begin with, will be in the CONNECTIONS AND DIRECTORIES section. Initially, this section will look like this: 

RealmID = 1
DataDir = "."
LogsDir = ""
LoginDatabaseInfo     = "127.0.0.1;3306;hellground;hellground;realmd"
WorldDatabaseInfo     = "127.0.0.1;3306;hellground;hellground;world"
CharacterDatabaseInfo = "127.0.0.1;3306;hellground;hellground;characters"
MaxPingTime = 30
WorldServerPort = 8085
BindIP = "0.0.0.0"
  • If this is your first server setup, the RealmID will remain 1. The server supports running multiple cores using the same realmd database. If you set up a second realm, the RealmID is 2, and so on.
  • Set the DataDir to the location of your client extracts. On Windows, this might be C:\HellGroundCore\Data. On Linux this will be the full path of $HOME/data (ex: /home/hellgroundcore/data.
  • Set the LoginDatabaseInfo, WorldDatabaseInfo, and CharacterDatabaseInfo to the appropriate setting. The hellgroundrealm.conf section above discusses the database line in detail, these will be set up in a similar fashion. The DATABASENAMEs will be different if you have decided to use other names.
  • For your first realm, the WorldServerPort should remain 8085. If you set up additional realms on the same server, this port will have to be different for each realm (second realm is 8086, etc)
  • Unless you have a good reason to change it, BindIP should stay 0.0.0.0. This tells hellground-core what IP address it should listen on. 0.0.0.0 means listen on any IP that exists on the machine. Most of us will not have multiple IP addresses on our machine, so this is an appropriate setting. If your machine has more than one network interface, and you only want the server to answer using one of them, set this to the IP address that is configured on the network interface that you want to have listening. If one or more of your network interfaces have more than one IP alias configured, you can further limit the server to listening on only one of those aliases by changing this to the alias address.


The next section that we need to look at is SERVER LOGGING. Initially, the section will look like this: 

LogSQL = 1
PidFile = ""
LogLevel = 1
LogFile = "Server.log"
ChatLogFile = "chat.log"
LogTimestamp = 0
LogFileLevel = 0
LogFilter_CreatureMoves = 1
LogFilter_TransportMoves = 1
LogFilter_VisibilityChanges = 1
WorldLogFile = ""
DBErrorLogFile = "db_errors.log"
CharLogFile = "characters.log"
CharLogTimestamp = 0
CharLogDump = 0
GmLogFile = "gm_commands.log"
GmLogTimestamp = 0
GmLogPerAccount = 0
RaLogFile = "ra_commands.log"
ArenaLogFile = ""
LogColors = ""
EnableLogDB = 0
DBLogLevel = 2
LogDB.Char   = 0
LogDB.GM     = 0
LogDB.RA     = 0
LogDB.World  = 0
LogDB.Chat   = 0
ChatLogs.Channel      = 0
ChatLogs.SysChan      = 0
ChatLogs.Whisper      = 0
ChatLogs.Party        = 0
ChatLogs.Raid         = 0
ChatLogs.Guild        = 0
ChatLogs.Public       = 0
ChatLogs.Addon        = 0
ChatLogs.BattleGround = 0
ChatLogTimestamp = 0

Fortunately, we don't need to make many changes here to get a running server. On Windows, you may not want to make any changes at all, at this point. On Linux, we will want to make a few minor changes to keep our $HOME directory neat and tidy.

  • First set PidFile to something appropriate. The recommended setting is $HOME/log/cored.pid, with $HOME being the full path (ex: /home/hellgroundcore/log/cored.pid). Why cored? Well, it's our core daemon, that's why (actually it is not technically a daemon, but that is a minor point - daemons are background processes, and these programs do not background).
  • Next, since us Linux types are so tweaky about such things, we want to change the LogFile to match the PidFile. Let's go with $HOME/log/cored.log (or /home/hellgroundcore/log/cored.log, for example).
  • Since we're testing a brand new core, let's get a bit more console log detail by setting LogLevel to 2.


Using VMaps

You will recall that earlier in the Guide we created our DBC, Map, and VMap files by extracting data from the client software. inorder to make use of the VMap (Visual Map) data, we have to enable a few settings in the PERFORMANCE SETINGS section of hellgroundcore.conf. If you choose not to use vmaps (it is optional, but highly recommended that you use them), then you will not need to make these changes.

The section that we are interested in initially looks like this: 

vmap.enableLOS = 0
vmap.enableHeight = 0
vmap.ignoreMapIds = ""
vmap.ignoreSpellIds = "7720"
vmap.petLOS = 0
vmap.totem = 0
vmap.enableIndoorCheck = 0


To enable vmaps:

  • Change the vmap.enableLOS, vmap.enableHeight, vmap.petLOS, and vmap.totem settings to 1.
  • To have players dismount when entering buildings, change vmap.enableIndoorCheck to 1. NOTE: This may cause them to dismount in some of the capital cities, and not be able to mount anywhere within the city.



At this point, we have made the minimal number of configuration file changes. As you start using the system, you are urged to read through the configuration options to see the rest of the possible changes.



The realmlist Table

Before we can start our new core, we need to make an entry for our realm in the realmlist table. This table is in the realm database.

We can accept most of the default settings for our new entry, but we will need to specifically set the name of our realm. As an example, we will call our realm HellGroundCore. We will also need to specify the server address. This can be a dotted-quad IP address, or a fully qualified domain name (FQDN). For our example, we will use a dotted quad address that is configured on our imaginary server, let's say 192.168.1.2.

Using whatever SQL tools works for you, INSERT a record into the realmlist table like this: 

    INSERT INTO `realmlist` (`name`,`address`) VALUES ('HellGroundCore','192.168.1.2');

You should use your real IP address and name! If you have a GUI SQL tool, feel free to use it.


Once we have the realm entry in our realmlist, we're ready to start our new server.
